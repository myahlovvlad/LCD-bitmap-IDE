# Current Codegen Audit

Phase 2A started with the existing generator unchanged. Phase 2B moves
production export buttons to `src/application/codegenFacade.ts`, which compiles
through `CompilerSourceSnapshot -> NormalizedCompilerIrV1 -> LoweredTargetIrV1
-> legacyCBackend`.

The legacy renderer generator remains in place as a compatibility oracle for
`npm run test:codegen-equivalence`.

## Scope

The current generator is renderer-owned. Production exports flow through
`src/renderer/core/ExportEngine.ts`, which delegates to
`src/renderer/utils/codegen.ts` and `src/renderer/utils/render.ts`.

The Phase 2A compiler layer must not change this path, generated C text,
generated binary bytes, packing, symbols, or schema-v5 project files.

## Implicit Assumptions

| Assumption | Current behavior |
| --- | --- |
| Display resolution | Default export is 128 x 64 from `DISPLAY_CONSTRAINTS`; custom width/height can be passed for selected screen exports. |
| Color depth | Monochrome 1-bpp only. |
| Packing | Vertical-LSB page packing. |
| Byte order | Pages first, then x columns, then bit 0 at the top row of each page. |
| Screen ordering | All-screen exports preserve the caller-provided `CanvasData[]`; UI callers build that from `project.screenOrder`. |
| State ordering | Runtime/FSM paths use `project.fsm.stateOrder`; C all-screen table stores state IDs from canvases. |
| Event ordering | Runtime and FSM UI use `project.fsm.eventOrder`; current C screen export does not emit events. |
| Transition ordering | Runtime uses `project.fsm.transitionOrder`; current C screen export does not emit transitions. |
| Locale selection | Text rendering chooses `object.text[language] || object.text.ru || object.text.en`. |
| Font source | Default bundled font unless caller passes a `FontRenderer` created from project-associated `fontGlyphs`. |
| Unsupported object behavior | Unknown object kinds are ignored by `renderCanvasObjects`; current known kinds are text, line, rect, icon, bitmap, special, invert. |
| Symbol collisions | `sanitizeSymbolName` normalizes identifiers but does not de-duplicate collisions. |
| Validation | Export blocking is handled by UI/application validation, not by the low-level generator. |

## Phase 2A Guidance

Phase 2A may introduce a renderer-independent `src/compiler` with a source
snapshot, normalized semantic IR, diagnostics, traceability, canonical
serialization and fingerprinting. It must leave the files above on the
production export path. Phase 2B can later compare IR-derived lowering against
the audited legacy path and switch production generation only after equivalence
is proven.
